# Contributing

Thank you for your interest in contributing.

## Scope

This repository is part of the **Kerio Connect Monitoring & Logging** project family. Contributions should keep documentation practical, reproducible, and safe for lab use.

## Ways to contribute

You can help by:

- reporting bugs;
- improving documentation;
- keeping `README.md` and `README.ru.md` aligned;
- suggesting new anonymization examples;
- fixing reproducibility issues;
- improving beginner-friendliness.

## Before opening a pull request

Please:

1. read the current `README.md`;
2. verify your change locally;
3. update documentation when behavior changes;
4. update `README.ru.md` when `README.md` behavior changes and translation is feasible;
5. keep examples realistic and safe to publish;
6. avoid unrelated cleanup in the same pull request.

## Documentation expectations

Documentation in this project family should be:

- written in English for canonical project files;
- mirrored in Russian for `README.ru.md` when feasible;
- practical and copy-paste friendly;
- reproducible without hidden assumptions;
- explicit about expected results and verification steps.

When changing documentation, please include:

- exact commands where possible;
- example input and output when relevant;
- a short verification method;
- notes about limitations or assumptions.

## Language policy

- `README.md` is the canonical English README.
- `README.ru.md` is the Russian quick-onboarding translation.
- `CHANGELOG.md` stays English-only unless the project explicitly decides otherwise.
- GitHub Release Notes stay in English.
- Russian documentation should preserve the behavior described in the English README and should not introduce separate behavior.

## Code and configuration changes

For code or configuration updates, please include:

- a short explanation of the change;
- expected behavior;
- sample input and output when relevant;
- version notes if compatibility is affected.

## Issues

When reporting a problem, please include:

- repository name;
- environment details;
- steps to reproduce;
- actual result;
- expected result;
- logs, screenshots, or sample data if useful.

## Sensitive data

Do not include:

- real credentials;
- private email data;
- non-anonymized production logs;
- proprietary binaries or artifacts that cannot be redistributed;
- private mapping files unless they have been reviewed and are intentional to publish.

## Pull request style

Please keep pull requests:

- focused;
- easy to review;
- documented;
- consistent with the repository's existing file naming and structure.
